Critical rule: No team member — designer, engineer, or QA — may accept or implement tasks directly from any stakeholder without PM approval. All requests must go through the PM.

2Feature Development Lifecycle

Every feature follows this exact sequence — skipping steps is not permitted:

  1. PM creates a PRD — defines scope, success metrics, and acceptance criteria using the standard template.
  2. Design creates Figma prototype — low-fi wireframes first, then high-fi designs. Reviewed by PM and Engineering before any code is written.
  3. Engineering breaks PRD into tickets — Frontend, Backend, and QA tickets created in the board with effort estimates.
  4. Sprint planning — tickets assigned, estimated, and committed to the sprint.
  5. Code pushed to Dev environment — QA executes test cases derived from the PRD.
  6. Staging deployment — PM or product team conducts User Acceptance Testing (UAT).
  7. Merge and deploy to Production — only after QA sign-off. Deployed via CI/CD pipeline.

7Incident Management

  • Bugs are detected via monitoring tools (e.g., Sentry, UptimeRobot) or user reports.
  • All bugs must be logged immediately to the ticketing system — never via chat or verbally.
  • Bug reports must include: recorded test steps, severity level. Screenshots alone are not sufficient.
  • Level 1 triage is performed by Engineering on-call (rotation-based).
  • If urgent: alert the Product team channel immediately and notify CTO/PM.

8Change Management

  • All production-affecting changes must have a PRD or a documented change log before work begins.
  • Relevant teams must be notified 24 hours before changes are pushed to production.
  • System downtime may only be scheduled during off-peak hours.
  • Emergency changes require explicit CTO or PM approval before proceeding.

9Release Management

  • Releases follow a fixed weekly cadence (e.g., every Monday at 4PM local time).
  • QA must sign off before any code is merged to production.
  • All releases are announced in the team communication channel and the changelog is updated.
  • Post-release: logs are monitored for early issue identification; QA confirms the deployed version.

11Communication & Task Intake

"If it's not in the backlog, it doesn't exist."
  • No team member may accept or implement tasks directly from external stakeholders without PM approval.
  • All requests must be submitted to the PM as a requirement or feature request.
  • The PM reviews, prioritises, and — if valid — creates a ticket in the project board.
  • Only after the ticket is documented and approved can it be assigned to a team member.
  • Every update must include the ticket link and clear context on what was done.
  • QA reports must include: list of tested tickets, test evidence (logs or recordings), and confirmation of retesting.

13Quality Assurance Standards

  • QA tests all features before merging to the staging environment.
  • Regression tests are run for every major feature — not just the new code.
  • Load testing is performed for all APIs before public releases.
  • Bug tracking is mandatory: every bug must have a ticket with status flow (To Do → In Progress → Done).
  • Automated testing is used alongside manual testing for comprehensive coverage.
  • QA signs off on the staging build before any production deployment.
